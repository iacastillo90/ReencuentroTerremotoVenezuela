Parte I — Ejecutiva

1. Resumen ejecutivo

Reencuentro Terremoto Venezuela («Juntos te encontramos») es una plataforma web ciudadana, gratuita y sin fines de lucro, creada para ayudar a las familias a reencontrarse tras los terremotos del 24 de junio de 2026. Centraliza fotos y videos de personas y animales —tanto de quienes están siendo buscados como de quienes ya fueron vistos o encontrados— y usa inteligencia artificial y geolocalización para sugerir coincidencias y acelerar el reencuentro.

El mensaje no es solo «Se busca»: también es «Estoy aquí», «Estoy vivo», «Estoy bien». «Cada historia importa. Cada reencuentro transforma.» La plataforma está pensada para funcionar incluso con muy mala conexión.

Visión en una frase
Un solo lugar, confiable y disponible offline, donde buscar sin fricción y reportar con responsabilidad, y donde una IA conecta a quien busca con quien fue encontrado —dejando siempre la confirmación final a un ser humano.

Los cuatro problemas que resuelve

ProblemaCómo lo resuelve
La información se pierde en redes y grupos de WhatsAppLa centraliza en una sola base de datos buscable
Buscar entre miles de reportes es lentísimoIA que sugiere coincidencias automáticas rankeadas por similitud
La conexión se cae constantementeFunciona offline: guarda y sincroniza al volver la señal
Las webs se saturan por el tráfico masivoInfraestructura serverless/CDN que escala sin colapsar

2. Contexto: qué está pasando en Venezuela

Cada decisión de diseño responde a la situación real del país (fuentes en el Anexo L).

🌎 El terremoto
El 24 de junio de 2026, dos terremotos sucesivos (7,2 y 7,5, con 39 segundos de diferencia) golpearon el norte de Venezuela, con epicentro en San Felipe / Yumare (Yaracuy). La Guaira fue lo más devastado (zona de desastre, +100 edificios colapsados) y el aeropuerto de Maiquetía sufrió daños. Las cifras escalaron a +1.400 fallecidos, +4.500 heridos y +50.000 desaparecidos. Es el sismo más fuerte desde 1900.
📶 La conectividad (el dato más decisivo de diseño)
La estatal Cantv concentra ~66% del mercado y +90% de líneas fijas (punto único de falla). En marzo de 2026 la conectividad nacional cayó al 61%, con episodios por debajo del 20%. Tras el sismo, miles de familias no lograban comunicarse. Por eso el enfoque offline-first no es un extra: es el núcleo no negociable.
🌐 Las plataformas que ya existen
venezuelatebusca.com y desaparecidosterremotovenezuela.com (+40.000 reportes), entre otras, ya operan pero se saturan y no cruzan datos. Nuestro diferencial no es otro formulario: es cruzar, deduplicar y aportar matching + censo. Se buscará colaborar, no competir.

3. Qué es, qué no es y qué queda fuera del MVP

Qué ESQué NO es
Herramienta ciudadana de ayuda humanitariaNo es un canal oficial ni reemplaza a Protección Civil / Cruz Roja
Canal para reportar hallazgos y registrar búsquedasNo confirma identidades por sí solo: lo valida una persona
Buscador con IA que sugiere posibles coincidenciasNo vende ni comparte datos con terceros
Base para un futuro censo humanitario exportableNo conserva los datos para siempre: caducan tras la emergencia
📦 Fuera del MVP (van a la hoja de ruta)
Donaciones, gestión de voluntarios, verificación de identidad, reconocimiento facial, bot de WhatsApp y la Biblioteca de emergencia. La ingesta/deduplicación de plataformas externas y el censo (lo más maduro del repo de Iván) también es Post-MVP: alimentará la plataforma más adelante (ver Anexo A y H).
🧭 Regla de oro
El sistema nunca afirma «encontrado» o «match confirmado». Muestra posibilidades rankeadas y deja que una persona confirme. Las plantillas jamás usan «confirmado», «encontrado» ni «match seguro».

4. El corazón del sistema: modelo bidireccional

La plataforma une dos lados y los cruza, en lugar de solo listar desaparecidos:

  • Lado Búsquedas («Se busca»): familias y diáspora registran a quién buscan.
  • Lado Hallazgos («He visto a alguien / Estoy aquí»): se reporta a quién se encontró o avistó.
  • Motor de matching: la IA compara ambos lados y avisa automáticamente a quien buscaba.

6. Recorrido por los módulos (propuestas de UX)

Las pantallas provienen de los mockups del equipo: ilustran la intención, no el diseño final pixel-perfect. Los prototipos finales por página acompañan este documento.

6.1 · Onboarding y bienvenida

Primer contacto cálido: la promesa («Cada historia importa. Cada reencuentro transforma.») y dos acciones claras: Buscar persona o Necesito ayuda. Carrusel introductorio de beneficios.

6.2 · Buscar persona (sin iniciar sesión) — foco del entregable

  • Buscador público, sin fricción: no requiere cuenta (en una emergencia, cualquier traba cuesta tiempo).
  • Búsqueda por descripción en lenguaje natural (IA semántica) y por voz.
  • Filtros combinables: categoría, zona/municipio, fecha, estado del caso.
  • Resultados en tarjetas ordenadas por similitud; ver perfil del caso.
  • Para menores: la ficha pública se restringe (ver §7 y Anexo F/G).
Detalle UX para menores (del documento de Búsqueda)
Al abrir la tarjeta de un menor se muestran características identificables (lunares, cicatrices, evento, edad) sin el nombre; en la misma pantalla se solicita correo, nombre, apellido, identificación y foto del solicitante, y se responde «vas a ser contactado por nosotros» sin exponer datos del menor.

6.3 · Reportar / Casos (con revisión del equipo)

  • Cuatro tipos: persona desaparecida, he visto a alguien (avistamiento), información anónima y otro.
  • Categorías: menores, adultos, adultos mayores y mascotas.
  • Video corto (15–30 s) recomendado + fotos; compresión en el teléfono.
  • Señas físicas, marcas/tatuajes, ropa, última ubicación (GPS) y fecha.
  • Descripción por texto o nota de voz (transcripción automática).
  • Cada reporte se convierte en un caso con estados: en búsqueda · en revisión · reencontrado · cerrado.
  • Código de reencuentro de 4 dígitos para cerrar el caso luego (incluso sin sesión).

6.4 · Mensajería

Canal interno para coordinar el reencuentro: bandeja de entrada, conversaciones, adjuntos y notificaciones. Mejora propuesta: enmascarar los datos de contacto hasta que ambas partes acepten, para proteger la privacidad y reducir el riesgo de extorsión.

6.5 · Mapa de reportes

Visión geográfica de la emergencia: mapa de calor, clústeres por ciudad, filtros (período, tipo, estado) y detalle por área. Para el MVP basta Leaflet + leaflet-heat (gratis). Cuidar no exponer ubicaciones exactas de domicilios para evitar saqueos: se ofusca a nivel de ciudad/zona.

6.6 · Perfil y configuración

Datos del usuario, privacidad y seguridad, sus casos y favoritos, y preferencias (modo claro/oscuro, accesibilidad).

7. Acceso, registro e inicio de sesión

Principio rector
Buscar nunca requiere cuenta; publicar o recibir alertas, sí. Da trazabilidad sin exponer al reportante, frena reportes maliciosos y habilita alertas y edición/cierre.

Niveles de acceso

NivelQuiénQué puede hacerRequisito
AnónimoCualquiera (incl. diáspora)Buscar · ver fichas (salvo menores) · mapa de calorNinguno
AutenticadoReportante / familiar / rescatistaPublicar caso · editar/cerrar · mensajería · alertasLogin
Verificado (post-MVP)Rescatista / tutor verificadoVer datos completos de menores de edadVerificación de identidad

Métodos de inicio de sesión (Supabase Auth, sin contraseña)

  • Google (OAuth): un toque; ideal para la diáspora y gama alta.
  • Teléfono + OTP: preferente por WhatsApp (aguanta 1 barra), respaldo SMS.
  • Email + enlace mágico: respaldo de bajo consumo de datos.
Por qué passwordless
Menos fricción bajo estrés y menor superficie de ataque. El borrador del mockup pedía «contraseña/OTP»; recomendamos passwordless (Google + OTP), coherente con Supabase Auth.

8. Resiliencia: funcionar con mala conexión

Con la conectividad bajo el 20%, la app no pierde ningún reporte aunque se vaya internet a mitad de camino.

  • Service Worker (Workbox): cachea la shell de la app (cache-first), instalable como PWA.
  • Cola offline (IndexedDB): si no hay red, el reporte se guarda local.
  • Background Sync: al volver la conexión, sube lo pendiente automáticamente.
  • Degradación del matching: sin red, búsqueda local aproximada hasta recuperar señal.

9. Lenguaje y tono del sistema

Cada mensaje a quien busca a un ser querido pesa. El tono equilibra claridad y sensibilidad humana.

  • Español neutro, claro y directo; una acción por mensaje.
  • Sensible, nunca clínico.
  • Sin certezas falsas: nunca promete encontrar a nadie.
  • Sin celebraciones inapropiadas ante un fallecimiento.
  • Reconocimiento del dolor en momentos críticos.
Ejemplo — posible coincidencia
«Encontramos un registro que podría corresponder. Esto es una posibilidad, no una confirmación; solo vos podés decidir. Para ver más detalles: [link]».
Ejemplo — reporte de fallecimiento
«Entendemos que esta noticia es muy difícil. Antes de registrarla, confirmá que estás seguro/a, ya que puede ser vista por familiares. ¿Confirmás? Sí / No».

10. Identidad visual y diseño

Marca Reencuentro Terremoto Venezuela con el lema «Juntos te encontramos». Logo: un corazón con la bandera y dos personas abrazándose. Estética de tema oscuro con acentos azul/morado, pensada para legibilidad bajo estrés y ahorro de batería en cortes eléctricos.

  • Accesibilidad AA/AAA, modo claro/oscuro, navegación por teclado y lectores de pantalla.
  • Responsivo (móvil primero).
  • Carga progresiva (skeletons) y manejo claro de errores offline.

Parte II — Anexos técnicos

Anexo A · Decisión de arquitectura (ADR)

El hallazgo que esta decisión resuelve
En los materiales coexistían tres visiones técnicas distintas. Sin unificarlas, el equipo construiría en direcciones incompatibles. Esta sección fija una sola.
VisiónStackFilosofíaEstado real
SPEC MVP v1.1 + modelo de datos 28/06Next.js + Supabase (Postgres + pgvector + PostGIS)Plataforma ciudadana bidireccional con matching, login, mensajería, PWADocumentado; por construir
Mockups + diagrama de flujoNext.js + Laravel + Node + Supabase + PythonIgual que el SPECEl propio SPEC pide simplificarlo a serverless
Repo de Iván (iacastillo90)React+Vite + Node/Express + MongoDB + Redis/BullMQAgregador/censo: scrapea y deduplica fuentes externas~70% back / ~80% UI, pero es otro producto

Decisión

Arquitectura canónica del MVP
Next.js + Supabase (serverless puro), construido en el repo pitonisaX/reencuentros-terremoto-vzla. El repo de Iván se usa como referencia de UI/páginas y, más adelante, como motor de ingesta/censo (Post-MVP).

Justificación

  • Un solo proveedor (Supabase = Auth + DB + Storage + Realtime + Edge Functions) → mínima operación para un equipo de voluntarios sin DevOps dedicado.
  • Postgres + pgvector + PostGIS resuelve búsqueda semántica + geográfica en una sola base, nativamente. MongoDB exigiría Atlas Vector Search (no implementado) y es peor para joins/geo del modelo relacional.
  • El modelo de datos más reciente (28/06) es relacional de 12 tablas con embedding_texto y embedding_imagen: confirma la dirección Postgres/Supabase.
  • Auth passwordless (Google + OTP) lista en Supabase.
  • PWA offline-first encaja con Next.js + Workbox.
  • El repo destino pitonisaX ya es Next.js 16 + React 19 + Tailwind 4 + Supabase: la base correcta ya existe (hoy es un scaffold).
Qué se aprovecha del trabajo de Iván
Su backend Node/Mongo es un excelente servicio de ingesta/censo: scrapers reales (USGS sismos, GDACS alertas, NASA FIRMS), deduplicación por idHash (SHA-256) + bulk upsert, y extracción de texto libre con LLM. Eso corresponde a Post-MVP 2 y convivirá como servicio aparte que alimenta la plataforma. No se descarta: se reposiciona.

Anexo B · Arquitectura técnica

Arquitectura serverless Supabase-céntrica: la base de datos (Supabase/PostgreSQL) es la única fuente de verdad. Se opta por serverless (sin servidor propio que mantener) para minimizar operación y costos.

Capas

CapaTecnologíaNotas

Resiliencia offline (PWA)

  • Workbox (Service Worker): cachea la shell de la app (cache-first).
  • Cola offline (IndexedDB / idb): si no hay red, el reporte se guarda local.
  • Background Sync: al volver la conexión, sube lo pendiente.
  • Degradación del matching: sin red, búsqueda local aproximada hasta recuperar señal.
💲 Costo estimado primeros 30 días
Entre $0 y ~$80 USD según volumen. El free tier de Supabase + Vercel + Cloudflare cubre lo básico; el único costo variable relevante es la API de Embeddings.

Anexo C · Modelo de datos

PostgreSQL en Supabase con pgvector (semántica) y PostGIS (geo). El modelo de datos del 28/06 define 12 entidades relacionales. Abajo, primero el panorama de tablas y luego el DDL del núcleo.

EntidadPropósito
usuariosCuentas (vinculadas a auth.users de Supabase); rol, verificación, moderación
ubicacionesPaís/estado/ciudad/municipio + lat/long + geografía (PostGIS)
reportesCaso bidireccional: tipo, categoría, estado, embeddings (texto e imagen), código de reencuentro
medios_reporteFotos/videos/audio del reporte (Storage); miniatura, duración, dimensiones
solicitudes_busquedaLo que alguien busca (con embeddings) para generar alertas y coincidencias
coincidenciasCruce reporte ↔ solicitud con puntajes (texto, imagen, geográfico) y estado
evidencias_coincidenciaExplicación de por qué algo coincide (trazabilidad de la IA)
historial_estadosAuditoría de cambios de estado de cada caso
contactos_casoRegistro de contactos/mensajería ligada a un caso
biblioteca_emergenciaHospitales/refugios/teléfonos (Post-MVP)
cola_moderacionRevisión del equipo antes de publicar
solicitudes_verificacionVerificación de identidad (rescatistas/tutores, Post-MVP)

DDL del núcleo (extracto)

create extension if not exists vector;
create extension if not exists postgis;

-- Usuarios (vinculado a auth.users de Supabase)
create table usuarios (
  id           uuid primary key references auth.users(id),
  nombre       text not null,
  telefono     text not null,            -- verificado por OTP
  email        text,
  rol          text default 'ciudadano', -- ciudadano | rescatista | verificado | moderador
  es_moderador boolean default false,
  consentimiento_at timestamptz,
  created_at   timestamptz default now()
);

-- Reportes/casos: un solo modelo para AMBOS lados (bidireccional)
create table reportes (
  id            uuid primary key default gen_random_uuid(),
  usuario_id    uuid references usuarios(id),
  ubicacion_id  uuid references ubicaciones(id),
  tipo          text not null,            -- persona_desaparecida | avistamiento | anonimo | otro
  categoria     text not null,            -- menor | adulto | adulto_mayor | mascota
  titulo        text,
  nombre        text,
  cedula        text,
  edad          int,
  genero        text,
  descripcion   text,
  marcas        text,                     -- tatuajes, cicatrices
  ropa          text,
  estado        text default 'en_revision', -- en_busqueda | en_revision | reencontrado | cerrado
  es_menor      boolean default false,
  visible_publicamente boolean default false,
  requiere_revision    boolean default true,
  embedding_texto  vector(1536),          -- búsqueda semántica
  embedding_imagen vector(512),           -- Post-MVP (facial)
  codigo_reencuentro char(4),             -- cierre offline del caso
  created_at    timestamptz default now()
);

-- Solicitudes de búsqueda (esperan alertas + coincidencias)
create table solicitudes_busqueda (
  id            uuid primary key default gen_random_uuid(),
  usuario_id    uuid references usuarios(id),
  ubicacion_id  uuid references ubicaciones(id),
  categoria     text,
  nombre_buscado text,
  descripcion   text,
  embedding_texto vector(1536),
  estado        text default 'activa',
  es_menor      boolean default false,
  created_at    timestamptz default now()
);

-- Coincidencias (cruce bidireccional)
create table coincidencias (
  id            uuid primary key default gen_random_uuid(),
  reporte_id    uuid references reportes(id),
  solicitud_busqueda_id uuid references solicitudes_busqueda(id),
  puntaje_total float, puntaje_texto float, puntaje_imagen float, puntaje_geografico float,
  estado        text,                     -- posible | probable | revisar
  created_at    timestamptz default now()
);

-- Índices clave

Nota sobre el esquema
El SPEC traía una versión simplificada de una sola tabla `reportes`. El modelo del 28/06 la normaliza (ubicaciones, medios, solicitudes, coincidencias, moderación…). Recomendamos adoptar el modelo normalizado como canónico y mantener vistas/funciones que simplifiquen las consultas más usadas.

Anexo D · Motor de IA y matching

MVP: búsqueda semántica de texto. Se genera un embedding de la descripción y se compara por similitud coseno con pgvector (índice HNSW, < 100 ms). El reconocimiento facial se difiere a Post-MVP.

Fórmula de score compuesto (adaptativo)

# MVP (sin facial): pesos ADAPTATIVOS segun señales disponibles
pesos = {'texto': 0.60, 'geo': 0.40}
disponibles = {k: w for k, w in pesos.items() if senal_existe(k)}
total = sum(disponibles.values())
score_total = sum(score[k] * (w/total) for k, w in disponibles.items())

# score_geo:   1.0 si <=1km | 0.7 si <=3km | 0.4 si <=5km | 0.0 si >10km
# score_texto: 1 - cosine_distance(embedding_busqueda, embedding_reporte)
# Post-MVP (con rostro): {'facial':0.65,'geo':0.25,'texto':0.10}

Regla hardcodeada
Las plantillas nunca usan «confirmado», «encontrado» ni «match seguro». La confirmación siempre la hace una persona.

Anexo F · Seguridad, privacidad y RLS

  • RLS: cada usuario solo edita sus reportes; datos completos de menores solo a rol verificado.
  • Moderación: los casos pasan por revisión del equipo (cola_moderacion) antes de publicarse.
  • Mensajería: enmascarar contacto hasta que ambas partes acepten.
  • Fotos: URLs firmadas con expiración; coordenadas ofuscadas a nivel de ciudad/zona.
  • Menores: ficha pública mínima; detalle solo tras verificación.
  • Fallecidos: fuera de búsquedas públicas; export anónimo.
  • Rate limiting + WAF (Cloudflare) en auth (anti-spam OTP) y en escritura.
  • Retención: PII y fotos se eliminan/anonimizan a los 90 días del fin de la emergencia. Sin uso comercial.
Términos y condiciones (mínimos legales)

La app no debe garantizar el hallazgo.

G.3 · Flujo de búsqueda semántica

  1. El usuario escribe/dicta: «mujer mayor, cabello canoso, suéter gris, vista en Altamira».
  2. El cliente llama a un endpoint público POST /buscar (Edge Function).
  3. El servidor genera el embedding del query y ejecuta similitud coseno con pgvector (filtros duros aplicados primero).
  4. Devuelve tarjetas rankeadas por score_texto (+ score_geo si hay ubicación).
  5. El usuario abre la ficha; si es menor, se activa el flujo restringido.

G.4 · API (Edge Function + RPC)

-- Función de búsqueda semántica (SQL/RPC) con filtros duros y geo opcional
create or replace function buscar_reportes(
  q_embedding vector(1536),
  f_categoria text default null,
  f_estado    text default null,
  f_punto     geometry default null,
  f_radio_km  float default null,
  lim int default 30
) returns table (
  id uuid, titulo text, categoria text, estado text,
  ubicacion_desc text, score_texto float
) language sql stable as $$
  select r.id, r.titulo, r.categoria, r.estado,
         u.ciudad as ubicacion_desc,
         1 - (r.embedding_texto <=> q_embedding) as score_texto
  from reportes r
  left join ubicaciones u on u.id = r.ubicacion_id
  where r.visible_publicamente = true
    and r.estado <> 'cerrado'
    and (f_categoria is null or r.categoria = f_categoria)
    and (f_estado    is null or r.estado    = f_estado)
    and (f_punto is null or ST_DWithin(u.geografia, f_punto, f_radio_km*1000))
  order by r.embedding_texto <=> q_embedding
  limit lim;
$$;

// app/api/buscar/route.ts  (Edge Function, pública)
import { createClient } from '@/lib/supabase/server'

export async function POST(req: Request) {
  const { texto, categoria, estado } = await req.json()
  // 1) embedding del query (proveedor en servidor; nunca exponer la API key)
  const q = await embed(texto)               // vector(1536)
  // 2) RPC de búsqueda semántica con filtros duros
  const supabase = createClient()
  const { data, error } = await supabase.rpc('buscar_reportes', {
    q_embedding: q, f_categoria: categoria ?? null, f_estado: estado ?? null, lim: 30,
  })
  if (error) return Response.json({ error: error.message }, { status: 500 })
  return Response.json({ resultados: data })  // tarjetas rankeadas por score
}

G.6 · Protección de menores (obligatoria)

Regla de producto
La ficha pública de un menor no muestra el nombre. Muestra solo características identificables (lunares, cicatrices, evento, edad aproximada). Para pedir contacto, el solicitante deja sus datos y recibe «vas a ser contactado por nosotros»; nunca se le expone información del menor. El detalle completo solo es visible a rol verificado.
  • RLS: las filas con es_menor = true exponen una proyección mínima a usuarios anónimos.
  • El formulario de solicitud crea un registro en contactos_caso / cola de verificación, no un acceso directo.

G.7 · Estados vacíos, errores y offline

  • Sin resultados: sugerir ampliar filtros / cambiar términos; ofrecer crear una solicitud de búsqueda con alerta.
  • Offline: búsqueda local aproximada sobre lo cacheado; banner claro de «sin conexión».
  • Accesibilidad: navegación por teclado, foco visible, lectores de pantalla, contraste AA/AAA.
