Reencuentros Terremoto Venezuela

Plataforma de código abierto que centraliza, depura y unifica reportes de personas desaparecidas y encontradas tras desastres naturales en Venezuela, con asistencia de inteligencia artificial y un enfoque de privacidad y seguridad por diseño.

📅 Actualizado: 29 de junio de 2026 🧩 Monorepo (front + back) ⚖️ Código abierto · repositorio público 🔒 Sin datos sensibles ni claves
Nota de seguridad. Este repositorio es público. Esta documentación describe nombres de variables, rutas y decisiones, pero nunca contiene secretos, claves ni PII (datos personales de víctimas). Los valores reales se gestionan fuera del repositorio.

1 Introducción y contexto

Durante un desastre, la información de personas desaparecidas se fragmenta entre redes sociales, mensajería y bases improvisadas, lo que genera duplicación, desinformación y pérdida de tiempo. La plataforma actúa como fuente única de verdad: consolida reportes de múltiples orígenes en un registro limpio por persona, y asiste la búsqueda con IA, manteniendo siempre a una persona en el bucle de decisión para los reencuentros.

Objetivos

  • Centralizar los reportes de personas y mascotas para que no se pierdan en el ruido.
  • Acelerar reencuentros con coincidencias asistidas por IA y verificación humana.
  • Vencer la baja conectividad (app ligera, offline-friendly).
  • Soportar alta demanda con infraestructura gestionada que escala sola.

Tipos de usuario

RolQué puede hacer
Ciudadano / familiar (user)Buscar, reportar, crear alertas, contactar de forma enmascarada.
Organización verificada (verifier)Acceder a datos sensibles (ubicación exacta, casos de menores) con registro de acceso.
Administrador (admin)Moderar, gestionar roles, aprobar verificaciones, confirmar coincidencias.

2 Arquitectura del sistema Arquitecto

Modelo de servicios desacoplados en la nube (PaaS / Serverless). No se mantienen servidores físicos propios: los proveedores gestionan parcheo, alta disponibilidad y escalado automático —clave en una emergencia donde el tráfico puede multiplicarse 10× en minutos—. La contrapartida es que la seguridad depende de configurar bien esos servicios (ver §12).

Capas

  • Presentación: SPA (React + Vite) servida por CDN, con soporte PWA.
  • API: servicio HTTP (Express) sin estado, escalable horizontalmente.
  • Procesamiento asíncrono: workers BullMQ para IA, scraping y sincronización.
  • Datos: base documental (MongoDB), caché/colas (Redis) y almacenamiento de objetos (S3).

Principios de diseño

  • Desacople de tareas pesadas: la IA y el scraping corren en workers, no bloquean la API.
  • Idempotencia en la ingesta: upsert por identificador único (idHash).
  • Abstracción de IA: una fábrica permite cambiar de proveedor por configuración.
  • Privacidad por diseño: la proyección de PII está centralizada (ver §11).

4 Modelo de datos DBA

Base NoSQL documental (MongoDB). Se eligió por sus esquemas flexibles (campos anidados como coordenadas dentro de lastSeen sin joins) y su soporte de índices geoespaciales.

Entidad central: UnifiedPerson

Cada documento representa a una persona consolidada a partir de uno o varios reportes.

{
  name, normalizedName,           // nombre y forma normalizada (búsqueda)
  idHash,                         // huella SHA-256 única → deduplicación
  externalIds: [{ source, id }],  // orígenes que reportaron a la persona
  status: missing|found|deceased|unknown,
  lastSeen: { state, municipality, description, date,
              coordinates: [lng,lat] },  // GeoJSON Point, índice 2dsphere
  age, gender, description, photoUrl,
  data:     { cedula_hash, ficha_url, origen, verificado_por },  // PII protegida
  metadata: { urgencyScore, confidenceScore, aiProcessed,
              auditStatus, source, reportedBy, createdAt, updatedAt }
}

Índices: idHash (único) · normalizedName + lastSeen.state · status + urgencyScore · lastSeen.coordinates (2dsphere) · externalIds.source + externalIds.id.

DBA · rendimiento. El conteo total (/persons/counts) se cachea en Redis 5 min para no ejecutar countDocuments sobre 50k+ documentos en cada visita (latencia ~5 ms). La paginación pública está acotada (máx. 200) para prevenir scraping masivo.

6 Procesos y algoritmos CS

Ingesta y deduplicación — Bulk Upsert con firma de identidad

  1. idHash: firma SHA-256 de fuente + id externo → identificador único y reproducible.
  2. Bulk upsert: bulkWrite([...], {upsert:true}) en lotes; si el idHash existe, actualiza; si no, crea.
  3. Resultado: decenas de miles de duplicados se consolidan sin sobrecargar la base.

Procesamiento con IA (asíncrono)

Texto libre (formulario o n8n WhatsApp/Telegram)
   → API responde 202 Accepted y encola (cola ia-process)
   → Worker → LLM extrae { nombre, edad, ubicación, ... }
   → reconciliation (>95% fusiona · 85–95% a auditoría)
   → upsert UnifiedPerson · metadata.aiProcessed = true
Regla (LOPNNA). No se envían datos de menores a APIs de IA externas.

Búsqueda y emparejamiento — humano en el bucle

Una alerta (SearchRequest) genera coincidencias (Match) por similitud de nombre (fastest-levenshtein). El sistema propone; una persona verificada confirma (posible/probable → confirmado) antes de avisar a la familia. Nunca se auto-notifica.

9 Módulo Buscar

Definición: permite localizar a una persona o mascota y, si no aparece, crear una alerta para que el sistema avise ante una coincidencia.

Flujo

Familiar → Buscar (nombre/zona/categoría de edad)
   ├─ ¿Aparece? → ficha pública → (adulto) contactar enmascarado
   │                            → (menor) comunicado: gestión por organización
   └─ ¿No aparece? → crear alerta (category, isMinor)
                      → matcher → Match → revisión humana → contacto seguro

Categorías y protección

La pantalla ofrece Adulto · Adulto mayor · Adolescente · Niño/a. Al elegir un menor (adolescente/niño), no se muestran resultados directos: aparece un aviso de protección y un flujo de "Solicitar búsqueda" gestionado por el equipo (ver §11).

10 Autenticación y control de acceso (RBAC)

Métodos

  • Google OAuth — verificación del id token en el servidor.
  • Correo / contraseña — registro e inicio de sesión; las contraseñas se almacenan con scrypt (sal aleatoria por usuario); el hash nunca se devuelve al cliente.

Roles

Middlewares requireUser, requireRoles([...]), requireAdminOrVerifier y requirePartnerApiKey. La interfaz refuerza la política como defensa en profundidad; la fuente de verdad es el backend.

Verificación de organizaciones

user  → POST /api/auth/verification-request (evidencia)  → 'pending'
admin → aprueba/rechaza → si aprueba: rol = 'verifier'

11 Privacidad y protección de menores LOPNNA

Enfoque Privacy by Design: la privacidad está integrada en cada decisión.

  • Proyección de PII centralizada (person-view.util): decide qué ve cada rol. Teléfonos, correos, cédula y ubicación exacta no se envían al público.
  • Ofuscación geográfica: coordenadas públicas a nivel de zona (~1 km), nunca dirección exacta.
  • Cédula como hash (cedula_hash), nunca en claro.

Protección de menores

ElementoPúblicoOrganización verificada
Que el caso existeSí (conteos/mapa)Sí
Nombre / foto / ubicación / contactoNo (caso protegido)Sí (con registro)
AcciónComunicado de gestiónBuscar, verificar, contactar

Implementación: toPublicPerson() devuelve un "caso protegido" cuando age < 18; el cliente lo replica con isProtectedMinor().

12 Seguridad Ciberseguridad

Controles de aplicación

  • Helmet (cabeceras), rate-limit global y de subida, hpp, límite de payload (1 MB).
  • CORS restringido por dominio (FRONTEND_URL).
  • JWT obligatorio: el servidor aborta en producción si falta el secreto; en desarrollo usa uno aleatorio por proceso (sin secreto hardcodeado).
  • Webhooks n8n validados con secreto compartido antes de encolar.
  • Subida de medios con autenticación y límite de tasa.
  • Validación de entrada con Zod; contraseñas con scrypt.
